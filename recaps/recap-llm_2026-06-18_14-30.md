# LLM Recap — Express API server + client mode detection
DATE: 2026-06-18 14:30
PROJECT: Ship Fees — Port of Beirut
STACK: Electron 29 + React + better-sqlite3 + Express 5

## TASK
Build a local HTTP REST API on port 3001 (Prompt 1) and a client-mode detection layer (Prompt 2) so the app can run as a thin client routing DB calls to a server PC over HTTP instead of hitting local SQLite.

## CONTEXT
Desktop billing app for Port of Beirut. All DB access goes through Electron IPC: renderer → preload (window.api) → main.js ipcMain.handle → handler functions in electron/database/handlers/. The REST API wraps those same handler functions so an external process or a second PC can call them. Client mode is triggered by a config.json file; without it the app runs normally (server mode). IPC handlers were NOT removed or replaced — client.js is a foundation for a future Prompt 3 that will wire IPC routing.

## CHANGES
**server.js — Prompt 1:**
- Generic handle(res, fn) helper wraps all handler calls: success → { success: true, data: result }, throw → { success: false, error }
- All handlers called with real signatures (verified against source): changePassword(userId, newPassword) not 3-arg; saveSession(data) single object for container+gc; getReport(year, month) not (month, year); receipt exports are getDataForReceipt/saveReceipt/softDelete not getData/save/delete; checkHasRecords not checkRecords; berthing.softDelete takes 3rd arg opts
- startServer() guarded by serverInstance singleton to prevent double-start
- Settings handler loaded from ./handlers/settings (verified path); fallback to ./database/handlers/settings if absent

**server.js — Prompt 2 login fix:**
- Login endpoint no longer uses generic handle(); manually calls authHandlers.login(), checks result.success, returns res.json({ success: true, data: result.user }) — user object is now directly in data, not nested inside { success, user }
- changePassword unchanged (returns { success: true }, no .user to unwrap)

**configStore.js:**
- getConfig() reads config.json once, caches in module-level _config
- Falls back to { mode: 'server' } on any read/parse error

**main.js:**
- const appConfig = getConfig() runs at module load time (before app.whenReady)
- if (appConfig.mode !== 'client') require('./server').startServer()
- ipcMain.handle('app:getConfig', () => ({ mode, serverUrl, token })) added after AI handlers

**client.js:**
- 37 functions; naming convention: {module}Verb for colliding names (berthingSoftDelete, receiptSoftDelete, containerSaveSession, gcSaveSession, etc.)
- apiCall() returns json.data — handlers return { success, data } which becomes the double-wrapped data field; callers get { success: true, data: rows } back, matching local handler behavior
- containerSaveSession(voyageNumber, lines, savedBy) packs to { voyageNumber, lines, created_by: savedBy } for server
- cmaGetReport(month, year) sends { month, year } named; server destructures by name so order mismatch is harmless
- changePassword(userId, currentPassword, newPassword) accepts 3 args but only sends { userId, newPassword } — handler doesn't validate current password
- fetch: uses globalThis.fetch ?? dynamic import('node-fetch') fallback

## WARNINGS
- Double-wrap: handle() wraps handler result in { success: true, data: <handlerResult> }. Handler result is itself { success: true, data: rows }. So apiCall returns { success: true, data: rows } — same shape as local call. If handler is changed to return raw data, client.js will break.
- server.js login endpoint is the ONLY endpoint not using handle(); if auth.js changePassword is ever updated to return { success, user } shape, it will also need manual unwrapping in server.js
- Do not move settingsHandlers to database/handlers/ — server.js and main.js both resolve it from ./handlers/settings
- express@5 was installed (not v4) — router behavior differs slightly; currently unused advanced features
- app.getPath('userData') in configStore.js: safe before app.whenReady() in Electron 29 / Node 20; do not call it from renderer process
